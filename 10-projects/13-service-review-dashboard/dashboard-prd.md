> 생성일: 2026-06-22
> 생성 방법: dashboard-prd skill (대화형)
> 작성자: 차재환 (교육팀)

## 1. 프로젝트 개요

### 배경 (Pain Points)
- 매월 경영진 보고를 위해 네이버·캐치테이블 리뷰 원본을 Gemini AI로 그때그때 분석 → **수작업 반복, 재현성·일관성 부족**.
- 13,000건이 넘는 리뷰를 사람이 다 읽을 수 없어, 브랜드/매장별 건전성을 한눈에 볼 수단이 없음.
- 불만 매장 요약·개선안을 매번 직접 작성 → 시간 소모.
- SL&C 자체 채널("고객의 소리")에는 위생·식재료·서비스 등 **크리티컬한 직접 접수**가 들어오지만, 별도로 관리·모니터링되지 않음.

### 핵심 의사결정
1. **(경영 모니터링)** 이번 달 어느 브랜드·매장의 리뷰 건전성이 좋아지고/나빠졌는지 보고, **집중 관리 대상**을 정한다.
2. **(개선 관리)** 불만율 상위 매장에 대해 무엇이 문제이고 무엇을 해야 할지 결정한다 (불만 요약·개선대책·완료방법).
3. **(교육/전사 시사점)** 반복되는 불만 토픽을 묶어 개선 방안·해결 완료 방안을 정의하고, 교육 커리큘럼·전사 정책에 반영한다.
4. **(리스크 대응)** SL&C 자체 채널의 크리티컬 건과 미답변 건을 즉시 파악하고 응대 완료를 관리한다.

### 목적
매월 1회, 흩어진 3개 채널(네이버·캐치테이블·SL&C 고객의 소리)의 고객 목소리를 브랜드/매장 단위로 통합·요약하여 경영진 보고와 서비스 개선 의사결정을 한 화면에서 지원한다.

### 핵심 가치
1. **재현 가능한 자동 집계** — 매월 같은 형식 데이터 → 같은 기준의 KPI (Gemini 즉석 분석 대체)
2. **읽지 않아도 보이는 불만** — 13,000건 텍스트를 AI가 감성·토픽으로 요약
3. **결정으로 직결** — 모든 화면이 "교육 대상 선정 / 개선 액션 / 리스크 대응"에 매핑
4. **크리티컬 직접 접수의 누락 방지** — SL&C 고객의 소리 응대 SLA 관리

## 2. 데이터 소스

### 소스 1: 네이버·캐치테이블 리뷰
- **위치**: `…/review_analysis_output/merged_reviews_YYYYMMDD_HHMMSS.csv` (예: `merged_reviews_20260622_145527.csv`)
- **형태**: CSV (UTF-8 with BOM)
- **규모**: 13,637행 × 6컬럼 (월 누적, 최근 6개월 위주)
- **생성**: 매월 네이버·캐치테이블 크롤링 → 동일 형식 CSV 자동 생성 (재현성 확보)
- **컬럼**: `store_name`, `review_text`, `rating`, `review_date`, `visit_date`, `source`
- **특이사항**:
  - **평점은 캐치테이블 312건(2.3%)에만 존재.** 네이버 13,325건은 평점 없음 → **칭찬/불만/중립은 AI 텍스트 감성분류로 산출**.
  - `store_name`은 `브랜드코드.지점명` 형식 (C/H/S/I/J).
  - BOM 오염: `﻿C.서래마을점`(79건)과 `C.서래마을점`(20건) 분리 → 정제 시 병합.
- **필요 컬럼**: 전부 + 파생(brand, sentiment, topic)

**브랜드 코드 매핑**
| 코드 | 브랜드 | 리뷰 수 |
|------|--------|---------|
| C | Chai (차이797) | 4,590 |
| H | 호우섬 | 4,568 |
| S | 서리재 | 2,575 |
| I | 이타마에 | 1,050 |
| J | 정육점 (바른고기 정육점) | 775 |

### 소스 2: SL&C 고객의 소리 (자체 채널)
- **위치**: `…/Downloads/customer_list_YYYYMMDD.xls`
- **형태**: 확장자는 `.xls`이나 **실제로는 웹 관리자 HTML 테이블 내보내기** (pandas `read_html`로 파싱)
- **규모**: 70행 × 14컬럼 (월 단위, 관리자 수동 내보내기)
- **컬럼**: NO, 매장명, 등록일, **구분**, **문의유형**, 작성자, 연락처, 이메일, 답변유형, 제목, **내용**, 등록일시, **답변**, **답변여부**, 답변일
- **특이사항**:
  - **이미 사람이 분류함** → AI 감성분류 불필요. `구분`(칭찬/불만족/의견) + `문의유형`(서비스/품질/기타) 그대로 사용.
  - **분류 매핑 확정**: 시스템 4분류 = 칭찬/불만/문의/기타. 데이터의 **"의견" → "기타"로 매핑**. "불만족"→"불만".
  - 처리상태 존재: `답변여부` = 답변완료 44 / **접수대기 25** → SLA 관리 가능.
  - `답변` 텍스트(점장 실제 응대)는 View 2의 개선대책/완료방법 **모범 사례**로 활용.
  - **PII 포함** (작성자·연락처·이메일) → 대시보드에서는 **마스킹**(예: 김○○, 010-****-7898) 후 표시.
  - 매장명은 풀네임("Chai797 BLACK 서래마을점") → 브랜드 식별 가능, 리뷰 데이터와 매핑 시 이름 정규화 필요.

## 3. 데이터 파이프라인

### 전체 흐름
```
[월 1회]
 네이버/캐치테이블 크롤링 ──► merged_reviews_*.csv ─┐
                                                   ├─► (1) 정제: BOM 병합, 브랜드코드 분리, 날짜 정규화
 SL&C 관리자 내보내기 ──► customer_list_*.xls ──────┘        │
                                                            ▼
                                          (2) AI 처리 (리뷰만):
                                              · 칭찬/불만/중립 감성분류
                                              · 불만 토픽/카테고리 태깅
                                              · TOP3 매장 불만 요약·개선대책·완료방법 생성
                                                            │
                                                            ▼
                                          (3) 집계 테이블 → Google Sheets (gws CLI)
                                                            │
                                                            ▼
                                          (4) Next.js 대시보드 (ISR) ◄── SL&C는 구분/문의유형 그대로
```

| 1 | 불만 토픽별 빈도 순위 (전 채널 통합) | 리뷰+VOC |
| 2 | 토픽 × 브랜드 매트릭스 (어느 브랜드에 어느 토픽이 집중되나) | 리뷰+VOC |
| 3 | 토픽별 월별 추이 | 리뷰+VOC |

**필터**: 기간(월) · 브랜드 · 채널(리뷰/VOC)
**활용**: 토픽 순위 → 다음 달 서비스 교육 커리큘럼 주제 선정.

### View 5: 매장 검색 · 드릴다운 — 특정 매장의 월별 추이와 불만 내용을 깊이 본다

> 매장이 매월 달라진다(신규 오픈). **그 달 리뷰가 1건이라도 들어온 매장은 자동으로 검색 대상에 포함**된다(브랜드 코드로 자동 분류, 별도 등록 불필요).

**표시 정보**
| 항목 | 출처 | 비고 |
|------|------|------|
| 매장명·브랜드 / 표시기간 총 리뷰·칭찬·불만·불만율 | 집계 | |
| 월별 표: 월 / 리뷰수 / 칭찬 / 중립 / 불만 / 불만율 | 집계 | 매장이 존재한 월만 행 생성 |
| 불만 리뷰 상세 (날짜 + 원문, 최신순) | 리뷰 | 검색 매장의 불만 전문 |

**상호작용**: 매장명 입력(부분 검색 가능 — 예: "스타필드", "강남") → 위 정보 즉시 표시.
**활용**: TOP3 외 임의 매장 점검, 신규 오픈 매장 초기 반응 모니터링, 특정 매장 보고 자료 추출.

## 5. 기술 스택

| 기술 | 용도 | 선택 이유 |
|------|------|-----------|
| Next.js + TypeScript | 앱 프레임워크 | ISR 캐싱, 월 1회 갱신에 적합 |
| Google Sheets (gws CLI) + ISR | 데이터 레이어 | 워크숍 표준, <50,000행, 갱신 월간, 시트 직접 수정 가능 |

| Python 전처리 (pandas) | 정제·AI 분류·집계 | CSV/HTML-xls 파싱, 감성·토픽 태깅, TOP3 요약 생성 → 시트 적재 |

> 데이터 규모(13,637행)와 월간 갱신 특성상 기본 워크숍 표준(Sheets) 채택. 실시간·동시성 요구가 생기면 Supabase로 승급.
